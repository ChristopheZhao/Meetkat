## 1. Requirement Analysis
- User intent:
  - 前两轮 Host-led 主线与 room-start owner correction 已完成，下一阶段先做 harness/governance 边界固化，再进入新的产品能力开发。
  - evaluation/validation harness 必须继续复用核心主线，但不能重新定义核心入口语义，也不能重新回流到产品主展示。
  - 当前步骤先建立 managed active plan，不直接开始编码。
- Scope in:
  - 盘点 repo 中仍可能把 harness/evaluation 语义带回 core、public contract、product surface 的入口、脚本、读模型、文档和测试。
  - 固化 explicit harness injection、product allowlist、canonical `room_start_contract` 的 repo-level boundary rules。
  - 增补 regression、文档与 operator workflow 约束，防止后续新需求静默重开 owner drift。
- Scope out:
  - 不重开 `PLAN-20260405-003` / `PLAN-20260407-004` 已关闭的问题。
  - 不新增产品能力或新的产品 IA。
  - 不把问题重新降级成 `preflight` taxonomy 优化。
  - 不把 `Control` 或 `Runtime` 扩回语义 owner。
- Acceptance criteria:
  - evaluation harness 是显式 governance/test capability，而不是默认产品/核心路径。
  - core/public contracts 继续围绕 canonical `room_start_contract`、machine-known facts、和通用 operator inputs 组织。
  - product surfaces 默认不能展示 validation-specific contracts。
  - 文档、脚本和回归测试让上述边界可审计、可回归、可复查。
- Architecture constraints:
  - 必须遵守 `docs/tech/MAS-harness-architecture-baseline.md`。
  - 必须遵守 `docs/tech/P1-host-led-meeting-topology-baseline.md`。
  - 必须保持 `room_start_contract` 是唯一 canonical room-start semantics artifact。
  - harness 只能通过显式 `operator_context`、smoke/evaluation adapters 复用主线。
  - 不允许新增 validation-specific product panels 或 core entry scopes。

## 2. Problem Statement
- 急性 owner drift 已修复，但 repo-level governance 仍偏软：
  - 脚本、文档、测试和命名仍可能模糊 evaluation harness 到底是 governance-only capability，还是产品化入口的一部分。
  - 未来 cleanup 或 feature work 可能通过改名字段、默认注入、便利脚本，再次把 validation semantics 带回 core/public contracts。
  - 当前边界更多依赖近期实现记忆，而不是 durable repo-wide guardrails。
- 本计划关闭的问题：
  - harness/evaluation semantics 通过文档、脚本、read model、默认值或回归缺口重新回流到 core/product。
  - renamed or indirect re-entry 缺少负例回归。
  - operator/evaluation workflow 和产品 workflow 仍可能混讲。
- 本计划不处理：
  - Host-led meeting architecture 重设计。
  - 新的 entry mainline。
  - `preflight.py` taxonomy 扩张或 room-start semantics 范围扩张。

## 3. Execution Sprints

### Sprint H1: Boundary Inventory Freeze
- What:
  - Inventory 当前所有 harness/evaluation touchpoints，覆盖 orchestration、runtime、frontend、scripts、tests、docs。
- Why:
  - 不先显式列出 leak surface，就容易在 cleanup 时误伤主线或漏掉隐性回流点。
- Done signals:
  - 存在一份 leak register，并把每个 touchpoint 标为 `allowed`、`must tighten` 或 `out of scope`。
  - inventory 明确区分 generic operator inputs 和 evaluation-only acceptance semantics。
- Evidence checklist:
  - 搜索或记录覆盖 `operator_context`、smoke/evaluation scripts、frontend read models、docs terminology。
- Negative cases:
  - 不做 inventory 就直接 cleanup。
  - 把所有 operator-facing context 都错判成 harness-only，破坏合法产品流程。
  - 重新讨论已经关闭的 room-start owner 问题，而不是盘点剩余 leak path。

### Sprint H2: Core And Contract Boundary Hardening
- What:
  - 收紧 core/public contracts，确保 evaluation semantics 不会被误当成 canonical room-start 或 runtime/control inputs。
- Why:
  - 当前主要风险不是架构方向错误，而是通过命名、默认值、adapter、read model 发生 accidental re-entry。
- Done signals:
  - `room_start_contract` 继续是唯一 canonical room-start artifact。
  - `Runtime` / `Control` 继续只消费 declarative contract outputs 和 machine-known facts。
  - harness-specific semantics 如仍需要，只能是显式 side-channel input，而不是 public/default core semantics。
- Evidence checklist:
  - 不存在新的 validation-specific `entry_scope`、gate owner、或默认 runtime injection path。
  - 保留的 harness adapter 都是 explicit、non-canonical、non-default。
- Negative cases:
  - 通过 renamed draft fields、default operator context、或 helper inference 重新引入第二 owner。
  - 把 semantic ownership 移回 `Control` 或 `Runtime`。
  - 继续把 `preflight.py` taxonomy 当成架构面增长。

### Sprint H3: Product Surface And Operator Flow Hardening
- What:
  - 确保产品页面、read models、以及 operator-facing flows 只暴露 product-safe 的 room-start 和 meeting outcome semantics。
- Why:
  - relabeled harness panels 很容易通过 UI 或 operator convenience flow 回流。
- Done signals:
  - product surfaces 继续只暴露 allowlisted、product-safe contract/read-model sections。
  - evaluation-specific acceptance structures 只留在 harness/operator/debug channels。
  - docs 和 scripts 清楚区分 product workflow 与 evaluation workflow。
- Evidence checklist:
  - `home/room/results` 与 shared frontend helper 持续落在同一 allowlist boundary 上。
  - operator/smoke scripts 不暗示 validation contracts 是正常产品 UI fact。
- Negative cases:
  - 用新标题重新带回 validation-specific cards。
  - operator convenience flow 静默变成 product default。
  - 代码边界收紧了，但 docs/scripts 仍在教错误工作流。

### Sprint H4: Regression And Governance Closure
- What:
  - 增加 durable tests、notes、closure criteria，让 harness boundary 在后续窗口里仍可执行。
- Why:
  - 没有 repo-level regression，下一个 feature sprint 很容易无意中 undo 当前分离。
- Done signals:
  - 有明确的 harness-to-core 与 harness-to-product leak-path negative-case regressions。
  - 边界规则被写到未来贡献者看得见的地方。
  - 本计划完成必须依赖代码/测试/用户确认，而不是 prose-only cleanup。
- Evidence checklist:
  - 覆盖 renamed/indirect validation semantics re-entry 的 regression。
  - 有 plan-linked notes 或文档定义 allowed harness usage。
- Negative cases:
  - 继续依赖一次性 review 记忆。
  - 没有 durable guardrails 就宣称完成。
  - 只是把 harness semantics 改名，而不是消除 leak path。

## 5. Test and Acceptance
- Architectural acceptance:
  - `Orchestration` 继续通过 `room_start_contract` 持有唯一 room-start semantics owner。
  - `Runtime` 与 `Control` 继续不是语义 owner。
  - evaluation harness 继续是 explicit、reusable、non-default 的 governance/test capability。
  - product surfaces 不重新承载 validation-specific 主展示。
- Verification:
  - 后端定向测试覆盖 entry scope、runtime gating、canonical contract persistence。
  - 前端 contract/read-model 回归覆盖 product allowlist behavior。
  - 脚本与文档检查证明 evaluation workflow 是显式的、非默认的。
- Negative cases:
  - validation-specific semantics 通过 renamed fields、panels、或 default script injection 回流。
  - `preflight` 再次长回架构概念，而不是保持实现容器。
  - 后续 feature 在 tests 通过的情况下，仍能把 harness semantics 泄漏到 product 或 core。
