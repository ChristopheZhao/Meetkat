## 1. Requirement Analysis
- User intent:
  - 当前窗口已确认 post-B5 的偏差不在“是否存在会前 contract assessment”，而在 room-start 语义 owner 被拆裂。
  - validation 应落在 evaluation/test harness，而不是继续在 agent 核心里长出第二条入口主线。
  - 编排决策必须由 agent-owned `Orchestration` 承担；系统代码只能保留声明式 contract plumbing、machine-known facts 注入、gate 执行与持久化。
  - 需要把这轮纠偏独立成新计划，并交给新窗口继续推进。
- Scope in:
  - 纠正 room-start contract semantics 的 owner 漂移
  - 收敛 `preflight.py` 的规则化语义 owner 趋势
  - 收敛 runtime/control 与 orchestration 在 room-start contract 上的边界
  - 收窄 product surface 对 validation-specific acceptance 语义的直接消费
  - 继续把 smoke/evaluation 语义固定在 harness 层
- Scope out:
  - 不回退 B1-B5 已完成的 Host-led meeting 主线
  - 不重新讨论 fixed rounds / fixed roster / fixed order 旧路径
  - 不把 `Control` 扩回会议语义 owner
  - 不在本计划内重做 PRD 或新增产品 IA
- Architecture constraints:
  - 必须遵守 `docs/tech/MAS-harness-architecture-baseline.md`
  - 必须遵守 `docs/tech/P1-host-led-meeting-topology-baseline.md`
  - room-start 产品主线只有一条，不允许再长出 validation-specific entry mainline
  - room-start contract assessment 只能是 `Orchestration` 的会前 capability，不是层，不是独立工作流 owner
  - `Runtime` 只做依赖组装、machine-known facts 注入、持久化与驱动
  - `Control` 只做 gate，不解释 contract semantics
  - evaluation/smoke/acceptance harness 只能复用同一会议主线，不得反向定义核心入口语义

## 2. Problem Statement
- 当前偏差的根因：
  - room-start 语义 owner 被拆裂为三方：
    - planner/open questions
    - `preflight.py` 的规则分类器
    - harness 注入的 validation contract
- 当前最明显的越界表现：
  - `preflight.py` 用 taxonomy/heuristics 决定 hard vs contextual vs missing operator inputs
  - runtime 在 planner 之后插入规则 adjudicator，再以其结果决定 room start
  - frontend product surface 继续直接展示 validation-specific acceptance contract
- 本计划关闭的问题：
  - 把 room-start contract semantics 收回到单一 agent-owned orchestration owner
  - 把系统代码收回到声明式 contract plumbing / gate / persistence
  - 把 validation-specific 语义从产品面与核心 owner 中收回到 evaluation/test harness

## 2.1 C1 Owner Split Table
- Canonical architecture concept:
  - `room_start_contract`
- Definition:
  - `room_start_contract` 是唯一 canonical 的 room-start semantics artifact。
  - 它由 `Orchestration` 内部的 contract-assessment capability 产出，并在 `planning.completed` 中持久化。
  - C2 可以调整具体代码字段名或实现位置，但收敛后必须只剩一个 canonical persisted artifact，供 `Runtime` / `Control` / product surface / harness 读取。
- Minimum semantics carried by the canonical artifact:
  - `room_start_ready`
  - `missing_operator_inputs`
  - `contextual_open_questions`
  - `known_context` 或同等 machine-known fact 引用
- Explicit non-goals for the canonical artifact:
  - 不承载 validation-specific acceptance taxonomy
  - 不承载 smoke/browser transport verification panels
  - 不承载第二条产品入口叙事

| Area | Single owner | Canonical artifact or output | Non-owner allowed work | Must not own or redefine |
| --- | --- | --- | --- | --- |
| Room-start semantics | `Orchestration` contract-assessment capability | `room_start_contract` | `Runtime` 传入 machine-known facts 并持久化；`Control` 读取 gate 信号；product 只读展示 | planner raw `open_questions`、`preflight.py` taxonomy、harness acceptance contract 并列充当 owner |
| Requirement brief and raw unknowns | requirement planner within `Orchestration` | `MeetingBrief` with raw `open_questions` | contract-assessment capability 读取这些 raw observations 并归一化进 canonical contract | 直接拿 raw `open_questions` 作为 room-start gate 或产品合同真相 |
| Machine-known facts | `Runtime/Harness` | `runtime_context` 或同等 declarative fact payload | 向 contract-assessment 提供 planner/executor target、bootstrap readiness、guardrail metadata | 解释 “哪些未知是 hard prerequisite / contextual question” 或直接生成 room-start semantics |
| Gate execution | `Control` | allow or block decision based on canonical contract | 消费 `room_start_ready`、缺口数量或等价 gate 字段 | 解释 contract semantics、维护 blocker taxonomy、把 gate 结果升格为语义 owner |
| Validation and evaluation defaults | `Governance/Harness` | harness-only acceptance inputs passed via explicit `operator_context` | 复用同一 room-start 主线做 smoke or evaluation | 定义核心 entry semantics、默认注入成为 canonical contract、长期沉淀到产品主展示 |
| Product surface | product read model and pages | 通用 room-start contract plus room status and conclusion | 展示 `missing_operator_inputs`、`contextual_open_questions`、通用 known context | 继续展示 validation-specific `success_criteria` / `validation_scenario` / `transport_contract` / `projection_contract` / `evidence_contract` / `conclusion_contract` 作为主展示 |
| `preflight` 这一实现名词 | no architecture owner; implementation container only | 可保留为代码文件或 helper 容器 | 承载 schema conversion、payload assembly、small helper logic | 被叙述成架构层、第二个语义 owner、第二条入口工作流 |

## 2.2 C1 Implementation Consequences
- C2 必须把 `planning.completed` 收敛到单一 canonical `room_start_contract`，而不是同时依赖 raw `open_questions`、rule taxonomy、和 harness contract。
- `src/decision_room/orchestration/preflight.py` 若保留，只能作为实现容器；其 taxonomy/heuristics 不得继续承担核心语义 owner。
- `src/decision_room/runtime/room_runtime.py` 只能把 planner 输出、machine-known facts、operator context 组装给 orchestration-owned contract-assessment capability，再读取 canonical contract 执行 gate。
- `src/decision_room/orchestration/room_executor.py` 只允许消费通用 canonical contract 与 machine-known facts，不再直接把 validation-only acceptance lists 当作会议 prompt 的常驻 known facts。
- `frontend/src/routes/home.tsx`、`frontend/src/routes/room.tsx`、`frontend/src/routes/results.tsx` 只能消费通用 room-start contract、缺失输入、contextual questions、room status、conclusion contract。

## 3. Execution Sprints

### Sprint C1: Owner Boundary Freeze
- What:
  - 冻结 room-start contract assessment 的抽象边界与 owner 分工
- Required deliverable:
  - 先产出一个明确的 `room-start owner split table`，至少覆盖：
    - 哪个单一 contract artifact 代表 `Orchestration` 拥有的 room-start semantics
    - planner/open questions 在其中只是输入，还是已经成为 owner 输出
    - `Runtime/Control` 只保留哪些 declarative contract plumbing、machine-known facts 注入、gate 与持久化职责
    - 哪些 validation/evaluation 语义只允许留在 harness/governance
    - product surface 允许展示什么、禁止展示什么
- Why:
  - 不先锁清 owner，后续实现很容易继续往 `preflight.py` 堆规则
- Done signals:
  - 文档和计划统一表达：
    - `Orchestration` 是 room-start semantics owner
    - `Runtime/Control` 不是 semantics owner
    - harness 不定义产品入口主线
  - `room-start owner split table` 已写入当前 active plan，并成为 C2-C4 的前置执行门槛
- Evidence checklist:
  - 诊断结论进入 active plan / handoff / 架构文档引用链
  - owner split table 明确标出：
    - 单一 canonical room-start contract artifact
    - planner/open questions 与 canonical contract 的关系
    - harness-only semantics 与 product-allowed surface 的分界
- Negative cases:
  - 继续把 `preflight` 当成独立工作流模块叙述
  - 继续把 “缺什么信息/是否 blocker” 归给 runtime/control
  - 在 owner split table 未冻结前直接进入 C2/C3 代码收缩

### Sprint C2: Core Contract Plumbing Reduction
- What:
  - 收缩核心实现中由规则代码承担的 room-start 语义裁定
- Why:
  - 当前 `preflight.py` 已经接近第二个语义 owner
- Done signals:
  - 规则代码只保留 schema/plumbing/gate 所需最小逻辑
  - room-start semantics 只由单一 orchestration-owned contract-assessment capability 输出
  - planner/open questions 只作为该 capability 的输入，不再与 canonical contract 并列充当语义 owner
- Evidence checklist:
  - `preflight.py` 不再维护大块 validation taxonomy 作为主要语义来源
  - runtime 不再把 planner + rule-adjudicator 串成双 owner
- Negative cases:
  - 继续新增 `provider_identity` / `success_criteria` / `transport_contract` 之类规则分类
  - 继续让 runtime 在 planning 之后插一个规则化语义裁判
  - 仅通过重命名/搬移 `preflight` taxonomy 就宣称 owner 已收敛
  - 同时保留多个 orchestration 输出并列定义 room-start semantics

### Sprint C3: Product Surface Contraction
- What:
  - 收窄产品页面对 validation-specific acceptance 语义的展示与依赖
- Why:
  - 当前 `home/room/results` 仍在产品化 harness 语义
- Done signals:
  - 产品页面只展示通用 room-start contract / missing inputs / room status
  - validation-specific acceptance panels 不再作为产品页主展示面
  - 不允许以改名后的摘要卡片或等价面板继续产品化 harness-only acceptance semantics
- Evidence checklist:
  - `home/room/results` 不再直接渲染 `validation_scenario` / `projection_contract` / `evidence_contract` / `conclusion_contract`
  - `home/room/results` 的允许展示面只保留：
    - 通用 room-start contract
    - missing operator inputs / contextual open questions
    - room status / conclusion contract
- Negative cases:
  - 结果页继续显示 smoke acceptance contract
  - 首页继续把 validation semantics 当主入口面板
  - 仅更换字段名或面板标题，但继续展示 harness-only acceptance contract

### Sprint C4: Harness Rebinding And Regression
- What:
  - 固定 harness/core/product 的新边界并补回归
- Why:
  - 不把 validation-specific 默认合同再回流进核心 owner
- Done signals:
  - smoke/evaluation harness 通过显式 `operator_context` 复用主线
  - 回归能防止 validation-specific scope/contract 再回流到 core
  - 回归能证明 harness-only acceptance semantics 不会重新成为 canonical room-start contract 或产品页主展示
- Evidence checklist:
  - smoke/test 仍可验证真实路径
  - core 继续拒绝 validation-specific core entry scope
  - regression 覆盖：
    - harness `operator_context` 不会生成新的 core entry semantics owner
    - product read model 不会因为 harness context 再次长出 validation-specific panels
- Negative cases:
  - harness 为了方便又重新引入核心 `entry_scope`
  - 产品页或 runtime 再次成为 harness contract 的长期 owner
  - harness 通过“默认 operator context”间接重新定义核心入口语义
