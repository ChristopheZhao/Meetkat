## 1. Requirement Analysis
- User intent:
  - 当前 P1 会议内核不能继续沿固定 `4` 轮、固定 `host/pro/con/recorder`、固定发言顺序推进。
  - 需要先定一个符合 multi-agent 初衷的现阶段技术方案，再继续实现。
  - 一期可以从中心化 orchestrator 起步，但不能退化成规则驱动工作流。
  - `MeetingPlanner/RolePlanner` 应前置为会前工作流；`Host` 应为唯一常驻 active agent；specialists 应按议题动态加入；`Recorder` 优先降为 memory-backed synthesis capability。
- Scope in:
  - 会前规划工作流：`RequirementPlanner -> RolePlanner -> RoleValidator -> AgentFactory`
  - 会中 `Host-led moderated hub-and-spoke topology`
  - 动态 specialist 实例化、加入与退出
  - `Recorder` 下沉为 synthesis capability 的 contract
  - `Control-Orchestration` 边界修订
  - 通用 pre-room preflight diagnostic/gate 能力
  - 基于新主线的真实联调与浏览器验证恢复条件
- Scope out:
  - 不覆盖分布式 agent mesh。
  - 不覆盖完全自由群聊。
  - 不覆盖生产级治理体系增强。
  - 不覆盖音视频和高并发基础设施。
  - 不把 validation/evaluation-specific entry logic 做成产品主入口或 MAS 核心架构的第二条主线。
  - 不把 smoke/acceptance 默认合同长期固化在产品入口语义中。
- Architecture constraints:
  - 必须遵守 `docs/tech/MAS-harness-architecture-baseline.md`
  - 必须遵守 `docs/tech/P1-host-led-meeting-topology-baseline.md`
  - `RoomEventJournal` 仍为唯一 SoT
  - `Control` 只做 lifecycle/retry/timeout/override/resume/fallback gate
  - `Orchestration` 必须负责谁发言、回答什么、是否继续、是否收敛、是否需要专家
  - `Runtime/Harness` 只驱动，不决策

## 2. Problem Statement
- 当前已确认失效的假设：
  - 固定 `4` 轮就是会议语义
  - 固定 `host/pro/con/recorder` 常驻就是多 agent
  - 固定 `host -> pro -> con -> recorder` 就是编排
  - 预算耗尽直接等于会议结束
  - `critical_decision_round=round>=4` 可以代表真实收敛阶段
- 本计划关闭的问题：
  - 用新的 P1 会议内核主线替换上述模板化假设，并恢复与真实联调相匹配的 agentic 会议骨架

## 3. Execution Sprints

### Sprint B1: Pre-room Planning Workflow
- What:
  - 落地 `RequirementPlanner -> RolePlanner -> RoleValidator -> AgentFactory`
- Why:
  - 不先把会前规划和会中 active agents 拆开，房间内 roster 会继续被固定常驻角色污染
- Done signals:
  - room start 前能产出 meeting objective、initial focus、constraints、open questions、candidate specialist roster
  - `MeetingPlanner/RolePlanner` 不再作为房间内常驻 speaker
- Evidence checklist:
  - 会前规划产物进入 `RoomEventJournal`
  - roster 由规划结果驱动，而非硬编码常量
- Negative cases:
  - 会前规划只生成展示文案，不真正驱动后续 roster
  - roster 仍然固定为 `host/pro/con/recorder`

### Sprint B2: Host-led Orchestration Rewrite
- What:
  - 重写会中编排，使 `Host` 成为唯一常驻 active agent，并由其决定下一位 specialist、handoff、challenge、summarize、conclude
- Why:
  - 当前固定发言顺序和固定 round curve 已经在替代编排层做语义决策
- Done signals:
  - `Host` 成为唯一常驻 active agent
  - specialists 可按议题动态加入/退出
  - 固定 `host -> pro -> con -> recorder` 顺序不再是系统主线
- Evidence checklist:
  - orchestrator 不再依赖固定 PARTICIPANTS 常量决定唯一会议流程
  - 下一位发言者由编排策略/Host 决定
- Negative cases:
  - `Control` 继续通过 round index 或 budget 推导语义下一步
  - 新 orchestrator 只是把旧的固定顺序换个文件

### Sprint B3: Memory-backed Synthesis And Conclusion Contract
- What:
  - 将 `Recorder` 下沉为 synthesis capability，并定义显式结论类型
- Why:
  - `Recorder` 不应默认是常驻 speaking agent，但其结构化产物必须保留为一等产物
- Done signals:
  - candidate decision / action items / open questions / agreement-disagreement summary 稳定进入 journal/projection
  - 会议结束必须输出显式结论类型：
    - `decision_reached`
    - `blocked_missing_information`
    - `human_escalation_required`
    - `runtime_failure`
- Evidence checklist:
  - `meeting.ended` 不再以“预算耗尽”直接充当会议完成语义
  - synthesis 产物可重放、可投影
- Negative cases:
  - 去掉显式 Recorder 后结构化产物退化成 UI 拼接
  - `budget end` 仍被直接当作会议结果

### Sprint B4: Control Boundary Rewrite
- What:
  - 把 `Control` 收回到运行底座职责：node/task/session state、retry、timeout、override、resume、fallback gate
- Why:
  - 当前 `Control` 已经越界承担部分编排语义
- Done signals:
  - `Control` 不再通过固定轮次、固定 decision round 推导语义阶段
  - `Control` 只消费标准化编排信号，不自发明会议语义
- Evidence checklist:
  - `max_rounds=4` 不再直接作为会议结束语义
  - `critical_decision_round=round>=4` 被移除或降级为非语义性 guardrail 信号
- Negative cases:
  - `Control` 继续决定谁发言、是否收敛
  - provider/routing helper 继续长出第二套编排语义

### Sprint B5: Re-validation Gate
- What:
  - 基于新主线恢复真实联调与浏览器验证
- Why:
  - 只有在新主线落地后，P1-3b/P1-3c 才重新有意义
- Done signals:
  - 新会议主线下可创建会议、动态加入 specialists、实时可见会议推进、显式结论落地
  - 浏览器验证不再依赖旧的固定 roster/固定 round 假设
- Evidence checklist:
  - room replay 能证明新主线是 agentic 编排而非模板流程
  - 结果页展示显式结论类型
- Negative cases:
  - 前端只是把旧 transcript 换皮
  - specialists 虽然存在，但不真正参与会议决策链

### Frontend Contract Guardrails
- backend authoritative contract / object:
  - `GET /api/rooms/{room_id}/snapshot` 是 room read-model 的唯一权威对象
  - `GET /api/rooms/{room_id}/replay` 与 `/api/rooms/{room_id}/events` / `/ws/rooms/{room_id}` 只承载同一 `RoomEventJournal` 事实流
- current frontend consumers:
  - `frontend/src/routes/room.tsx`
  - `frontend/src/routes/results.tsx`
  - `frontend/src/routes/home.tsx`
  - `frontend/src/lib/room-store.ts`
  - `frontend/src/lib/transport.ts`
- allowed fallback:
  - `WebSocket -> SSE` 仅允许作为 transport failover，且必须继续消费同一 room event stream + seq ack contract
- forbidden frontend authority-routing:
  - 前端不得从固定 `host/pro/con/recorder` 假设反推出 turn plan 或角色语义
  - 前端不得用 `consensus.reason`、budget exhaustion 或 transport state 充当会议结论语义
  - 前端不得在 `snapshot.current_turns` / `snapshot.conclusion_type` 已存在时自行拼装替代对象
- B5 前最小 UI contract:
  - room/results/home 必须直接消费 `planning_artifacts`、`current_turns`、`conclusion_type`、`conclusion_reason`
  - transport fallback 继续保留，但不得长成第二套 truth-routing
