## 1. Requirement Analysis
- User intent:
  - 先输出明确计划再开始实施。
  - 一期优先完成 MAS 算法设计实现、前端交互、后端 Demo 支撑。
  - 不把 FSM/规则作为技术核心；规则仅做门控护栏。
  - 后端与 MAS 解耦，保证后续生产化替换后端时算法层可复用。
  - 支持人类加入与 multi-agent 同场讨论；前期以 agent 自动化讨论为主。
  - 必须可见会议室实时讨论实况（发言流、回合状态、收敛进度）。
- Scope in:
  - P0: MAS 算法策略、模型路由、真实小流量试运行（non-mock core path）。
  - P1: 可演示前端 + 最小后端运行时 + 端到端会议闭环。
  - P1: 系统性评测与锁型判定。
  - P2: 生产级后端增强（分布式总线、鉴权、观测、SLA）。
- Scope out:
  - 一期不做重型企业治理系统、复杂规则引擎、多租户生产级完备能力。
  - 一期不追求音视频底层与高并发极限优化。
- Acceptance criteria:
  - 计划经用户确认后进入实施。
  - P0 跑通 Hybrid 主线闭环并完成评测方案准备。
  - P1 完成可演示闭环（可发起会议、可协作、可收敛、可导出）。
  - MAS 内核后续重构与实现必须遵守 `docs/tech/MAS-harness-architecture-baseline.md`：`Memory` 为基底、`state` 只是映射、`Runtime/Harness` 不是层、`Adapter` 不得膨胀为竞争主线。
  - P1 demo 不接受纯前端 mock；会议进行页必须消费真实 room runtime / room event protocol 事件流并反映真实会议状态。
  - requirement planner 默认 fail-fast；创建会议时若主 planner/上游模型失败，必须暴露真实错误，不允许隐式 `else -> fallback` 吞掉原因。
  - P1 完成锁型评测并冻结一期默认配置（模型与阈值）。
  - P1 demo 支持“人类加入并介入”与“agent 自动化主导讨论”两种模式。
  - P1 demo 可实时展示会议实况（角色发言流、当前轮次、收敛状态）。
  - P2 在不改策略接口前提下完成后端生产化替换。

## 2. Functional Decomposition
- Module A: MAS Core（模型驱动）
  - Agent 协作策略（handoff、角色租约、并行讨论）
  - 收敛策略（分歧指数 DI + 共识评分）
  - 模型路由（默认层/升级层/灾备兜底）
- Module B: Runtime & Communication（事件驱动）
  - WebSocket 主链路 + SSE 降级
  - 房间级事件通道、幂等、重放、断线恢复
  - 会话状态与执行日志
  - 供前端消费的 room state / transcript 派生视图
  - 人类介入命令回流（message / interrupt / vote / override）
- Module C: Product UX（会议体验）
  - 会议大厅、会议进行页、结果页
  - 人类介入（打断/追问/投票/裁决）
  - 实时会议实况面板（发言流、阶段、收敛指标）
  - 结果导出（Markdown/PDF）

## 3. Implementation Approach
- Data flow:
  - Client 发起 room -> Runtime 建立 room session -> MAS Core 消费上下文并发布协作事件
  - 协作事件经 EventBus 扇出到客户端与存储
  - ConsensusStrategy 周期计算收敛状态并触发 `consensus.check` / `meeting.end`
- API/contract changes:
  - 事件信封稳定字段：event_id/room_id/room_seq/producer_id/ts_ms/idempotency_key/payload
  - 关键事件：role.switch/handoff/consensus.check/human.override/meeting.end
  - 策略接口：PlanningStrategy/CoordinationStrategy/ConsensusStrategy/GuardrailPolicy/FallbackPolicy
- Migration or compatibility notes:
  - P0/P1 采用最小运行时实现；P2 仅替换后端基础设施，不改策略接口与事件稳定层
  - 保持 schema 版本化，增量升级

## 4. Technical Solution
- Key design choices:
  - 采用 Hybrid：MAS 智能核心 + Event Runtime + Guardrail Control Plane
  - FSM 降级为护栏层，不参与智能推理路径决策
  - Fallback 仅灾备，不作为常态 else 逻辑
  - MAS harness 总架构基线固定为：`Memory / Orchestration / Control / Governance`；`Runtime/Harness` 仅作为执行线束，`state` 仅作为 memory projection 术语存在
  - requirement -> meeting brief 的主路径必须由 agent planner 完成；任何 fallback 都只能显式开启，且 fallback stub 不承担语义理解
  - 模型策略：低成本主路径（Qwen3.5/MiniMax-M2.5）+ GLM-5 升级 + OpenAI 仅灾备
- Frontend stack decision (P1 baseline):
  - 渲染模型：CSR 单页应用；P1 不引入 SSR / RSC / BFF 绑定，保持前端与 room runtime 服务解耦
  - UI 框架：React 19 + TypeScript
  - 工具链：Vite 作为前端开发服务器与构建工具
  - 路由层：React Router v7 Data Mode，用于大厅 / 会议中 / 结果页路由与初始数据装载
  - 请求层：TanStack Query v5 负责 room 列表、历史会议、结果页、用户操作 mutation 等 HTTP 状态
  - 实时层：浏览器原生 WebSocket 主链路 + EventSource(SSE) 降级；前端只消费 room event protocol，不自定义第二套会议语义
  - 客户端状态：会议房间使用基于 `useSyncExternalStore` 的 room event store；传输回调使用 `useEffectEvent`；高频事件下对 transcript / panel 派生视图使用 `startTransition` / `useDeferredValue` 降低卡顿
  - 样式方案：CSS Variables + CSS Modules；先保证会议态信息密度与可读性，不引入额外 design system 框架依赖
  - 明确不选：Next.js / SSR-first 全栈方案；原因是一期目标是最小可控实时会议室和独立 room runtime，而非绑定框架服务端抽象
- Trade-offs:
  - 短期不做重治理可加速验证，但生产合规能力后置
  - 先用经验阈值可快速推进，但需要 P0/P1 数据校准
  - 选择 Vite SPA + React Router 可降低前期集成复杂度并保持后端边界清晰，但 SEO、服务端渲染和同构部署收益后置
- Risks and mitigations:
  - 风险: 讨论发散导致不收敛 -> 缓解: DI + 共识评分 + 人工仲裁阀
  - 风险: 模型可用性波动 -> 缓解: 重试/降载/同层替代后再灾备兜底
  - 风险: 后端过早复杂化拖慢进度 -> 缓解: 严格 P0/P1/P2 分阶段边界
  - 风险: 高实时事件流导致前端重渲染抖动 -> 缓解: room event store + 派生视图分层 + transition/deferred 渲染
