## 1. Requirement Analysis
- User intent (verbatim): "需要根据审核意见完成修正和重构，特别是 mas，是 multi-agent 系统，需要进行调研确定架构设计再开始，架构设计实现也可以参考 short-video-maker，只参考有价值的地方，直到实现预期可以进行 multi-agent 讨论决策会议室的预期"
- Trigger: deep review of commit 5f47149 found the shipped "centralized MAS decision room" is a deterministic scripted stub:
  - `CentralizedMeetingSupervisor._execute_role` emits 4 hardcoded paragraphs keyed by `role.stance`, no LLM call.
  - `should_end = round_index >= 5 and signals.confidence >= 0.68` reduces to a pure round counter because `confidence` is itself a linear function of `round_index`.
  - `CentralizedRequirementPlanner._central_constraints` pre-injects "Use a single supervisor to own agent assignment and convergence decisions" into the brief, then the supervisor "concludes" the same claim — circular.
  - Default `DECISION_ROOM_EXECUTOR` was flipped from `llm` to `centralized` so any environment without explicit override silently runs the stub instead of the real `LLMRoomExecutor` (1442 lines, fully wired with three-tier routing and signal-based consensus).
  - Frontend `room.tsx` reads `entry.artifacts.central_mas` directly, bypassing the `PRODUCT_OPERATOR_CONTRACT_SECTIONS` allowlist that PLAN-20260412-005 Sprint H3 was supposed to enforce.
- Scope in:
  - Replace the scripted supervisor with a real LLM-driven `LLMSupervisor` that owns role selection and per-agent assignment contracts.
  - Reuse existing `_generate_argument` specialist machinery and `HybridConsensusStrategy` signal gate.
  - Remove `CentralizedRequirementPlanner._central_constraints` answer injection.
  - Revert silent default-flip side effects: `centralized` executor mode requires real provider env, no silent stub fallback.
  - Route frontend `central_mas` reads through the planning-artifacts allowlist.
  - Add the H4-style negative regressions PLAN-20260412-005 still owes.
- Scope out:
  - Renaming `host` → `supervisor` across `LLMRoomExecutor` and 68 existing tests. The supervisor is implemented as a new orchestration component, not by relabelling the existing host LLM call site.
  - New product surfaces beyond what is needed to render real supervisor output.
  - Per-agent LLM policy (a la short-video-maker `llm_policies.yaml`). Documented as future work, not built now.
- Acceptance criteria:
  - Every specialist message in a centralized round comes from a real LLM call (provable: removing the provider env causes the executor to be unavailable, not produce stub output).
  - Supervisor role selection and assignment contracts are produced by a real LLM call against the snapshot, not derived from keyword matching.
  - Convergence ending is driven by `HybridConsensusStrategy` evaluating real signals, not by a counter dressed as a signal gate.
  - `central_mas` artifact only reaches the product UI through the planning-artifacts allowlist.
  - Backend regression suite covers: stub-vs-real divergence, planner does not inject answer-shaped constraints, convergence honors signals, `centralized` mode without provider env is unavailable.
  - All existing tests still pass; build green.
- Architecture constraints:
  - Must obey `docs/tech/MAS-harness-architecture-baseline.md` §2 Binding Rules: single SoT = `RoomEventJournal`; `Runtime` is not a layer; orchestration owns semantics; no second entry mainline; harness defaults must not固化 validation-specific semantics as product entry.
  - Must obey `docs/tech/centralized-mas-delivery-boundary.md` §3 layer split: `CentralizedMeetingSupervisor` owns role selection + assignment + convergence recommendation; specialists are typed role nodes; `RoomControlPolicy` keeps gating; `RoomRuntime` stays harness.

## 2. Architecture Direction (confirmed with user 2026-05-26)
- Supervisor topology: **Supervisor LLM directly emits assignment contracts** (user-selected option). The existing host LLM stays for backward compat in `executor_mode=llm`, but `executor_mode=centralized` uses a new `LLMSupervisor` that produces `assignment_contracts[]` directly (mirroring short-video-maker `_llm_decompose_tasks`).
- Stub disposition: **delete** `CentralizedMeetingSupervisor` class and its 4 stance-keyed templates. Keep `CentralAgentRole`, `AssignmentContract`, `SupervisorState` dataclasses as pure value types.
- Specialist execution: reuse `room_executor._generate_argument` / `_parse_argument_output` LLM machinery. Adapt the call site to consume `AssignmentContract` instead of `AgendaTurn`.
- Synthesis + consensus: reuse `_generate_synthesis` and `HybridConsensusStrategy`. Convergence remains real signal-gated.
- Brief planner: strip `_central_constraints` answer injection. `CentralizedRequirementPlanner` keeps a thin local-only path for offline dev (no provider env), but does not pre-inject the architectural answer.
- Default modes: `DECISION_ROOM_PLANNER_MODE` default reverts to `primary_with_fallback` (the original). `DECISION_ROOM_EXECUTOR` default reverts to `llm` (the original) — `centralized` becomes an explicit opt-in that requires provider env, returning `UnavailableRoomExecutor` when unconfigured (no silent stub).
- Frontend: `central_mas` is added to `PRODUCT_OPERATOR_CONTRACT_SECTIONS` allowlist OR read via a new `readCentralMasState()` helper in `planning-artifacts.ts`; `room.tsx` stops touching `entry.artifacts.central_mas` directly.
