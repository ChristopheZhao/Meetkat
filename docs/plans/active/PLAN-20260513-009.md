## 1. Requirement Analysis
- Trigger: PLAN-008 closed but left two SHAPING-grade rule-based residuals visible in the post-fix audit:
  - **Phase pacing is rule-derived**: `LLMRoomExecutor._phase_for_round` decides EXPLORE/DEBATE/SYNTHESIZE/DECIDE from `round_index`, `snapshot.candidate_decision`, `consensus.disagreement_index`, and transcript length. The synthesis LLM had no way to advance or pull back the pacing even when its judgment said the room should move on.
  - **HeuristicRolePlanner runs silently as fallback**: `PreRoomPlanningWorkflow.__init__` defaults `role_planner = HeuristicRolePlanner()` (keyword matching) when no LLM-driven planner is injected. The runtime never told the operator the meeting was running in a degraded role-selection mode.
- Scope in:
  - **C1**: `SynthesisOutput.recommended_next_phase` optional field; synthesis prompt asks for it; parser validates against {"explore","debate","synthesize","decide"}; agent.summary event carries it; projector persists it to `snapshot.recommended_next_phase`; `_phase_for_round` prefers it before the rule-based path.
  - **C2**: `PreRoomPlanningWorkflow.role_planner_kind` property exposes "heuristic" or "llm"; `RoomRuntime.runtime_readiness()` adds `role_planner_kind` + `role_planner_degraded` keys so the operator UI can render a degradation banner.
- Scope out:
  - Coordination strategy LLM recommendations (`HybridCoordinationStrategy` SPEAK/HANDOFF/CHECK_CONSENSUS FSM). Less impactful — coordination shapes journal events but not specialist prompts. Documented as PLAN-010 candidate.
  - Signal-math LLM rewrite (`_signals_for_round` / `_signals_after_round`). Synthesis already overrides via `should_end_meeting`. Documented as long-running debt.
  - Frontend banner for `role_planner_degraded`. Backend exposes the signal; frontend rendering deferred.
- Acceptance criteria:
  - Synthesis LLM can emit a valid next-phase recommendation and have it honored on the next round.
  - Invalid / empty recommendations are dropped — rule-based pacing takes over.
  - `runtime_readiness()` always includes `role_planner_kind` + `role_planner_degraded` keys.
  - All existing 110 tests still green; 5 new regression tests added.

## 2. Execution Sprints

### Sprint C1: LLM-recommended next phase
- `SynthesisOutput.recommended_next_phase: str = ""` added.
- `_build_synthesis_prompts` schema example now includes `"recommended_next_phase": ""` with an explicit prompt clause: "OPTIONAL: emit recommended_next_phase if you want to override the runtime's default phase pacing for the next round. Valid values: 'explore' (open up new angles), 'debate' (stress-test current claims), 'synthesize' (converge on a recommendation), 'decide' (lock the candidate). Leave it empty when the default rule-based pacing is fine".
- `_parse_synthesis_output` calls new `_coerce_recommended_next_phase` which drops invalid values to `""`.
- `LLMRoomExecutor.build_round` + `CentralizedMASExecutor.build_round` both add `recommended_next_phase` to the `synthesis_message.artifacts` bundle.
- `RoomOrchestrator.emit_round` lifts it from `synthesis_message.artifacts` into the top-level `agent.summary` event payload so the projector does not have to dig.
- `RoomSnapshot.recommended_next_phase: str = ""` field added; `RoomProjector._apply_agent_summary` reads the payload and persists when valid, clears when not (so a stale recommendation never leaks past one round).
- `LLMRoomExecutor._phase_for_round` first reads `snapshot.recommended_next_phase`; if it parses as a known `MeetingPhase`, returns it. Otherwise the original rule-based derivation runs.

### Sprint C2: role-planner degradation visibility
- `PreRoomPlanningWorkflow.__init__` distinguishes "role_planner injected" (kind="llm") from "default fallback to HeuristicRolePlanner" (kind="heuristic"). Exposed via new `role_planner_kind` property.
- `RoomRuntime.runtime_readiness()` reads the workflow property and adds `role_planner_kind` + `role_planner_degraded` keys to the readiness dict.
