## 2. Execution Sprints

### Sprint D1: Coordination LLM recommendation
- `SynthesisOutput.recommended_next_action: str = ""` added with valid set `{speak, handoff, check_consensus, end_meeting, noop}`.
- `_coerce_recommended_next_action` validates the field (strict whitelist + `_-/space` tolerance).
- Synthesis prompt schema + prose ask for it with explicit per-action semantics; LLM is told to leave it empty when the FSM choice is fine.
- `SynthesisMessage.artifacts` carries it; `RoomOrchestrator.emit_round` lifts it into the top-level `agent.summary` event payload; projector persists to `snapshot.recommended_next_action`.
- `RoomExecutor._coordination_for_turns` and `CentralizedMASExecutor._coordination_for_speakers` both accept `snapshot=`; honor the snapshot's recommendation before consulting the FSM.

### Sprint D2: LLMRolePlanner replaces HeuristicRolePlanner default
- New `LLMRolePlanner` class in `pre_room_planning.py`:
  - Takes the meeting brief + the existing `_ROLE_BLUEPRINTS` catalog.
  - Calls the configured provider with a JSON-mode prompt asking for 2-4 selections from the catalog (verbatim role IDs only).
  - Parses `selected_roles: [{role, join_reason?}]`; maps each to a `CandidateSpecialist` via the catalog blueprint, with an optional `join_reason` override.
  - On any transport / parse failure, falls back to `HeuristicRolePlanner` so a flaky provider does not block room creation.
- `LLMRolePlanner.from_mapping(env)` returns `None` when `MODEL_DEFAULT_*` env is missing; runtime keeps the heuristic fallback explicit (`role_planner_kind="heuristic"`).
- `PreRoomPlanningWorkflow.from_env` and new `from_mapping(env)` classmethod wire `LLMRolePlanner.from_mapping(env)` as the default role planner.
- `build_runtime_from_env` uses `PreRoomPlanningWorkflow.from_mapping(env_map)` so the runtime picks up LLMRolePlanner whenever provider env exists. `runtime_readiness.role_planner_kind` now reports `"llm"` in the default `.env`-configured path; `"heuristic"` only when no provider env is present.

### Sprint D3: Frontend role-planner degradation banner
- `RuntimeReadiness` TS type extended with `role_planner_kind?: string` + `role_planner_degraded?: boolean`.
- `home.tsx` reads `preflightReport.runtime_readiness.role_planner_degraded`; renders a Chinese warning banner inside the hero when true (orange-tinted). Banner instructs the operator to configure `MODEL_DEFAULT_SUPPLIER` / `MODEL_DEFAULT_MODEL` to upgrade to LLM-driven selection.
- New `styles.warningBanner` CSS class.

### Sprint D4: Room UI surfaces LLM recommendations
- `RoomSnapshot` TS type extended with `recommended_next_phase?: string` + `recommended_next_action?: string`.
- `room.tsx` Central MAS panel now shows a "下一轮 LLM 建议" sub-section with phase and action when either is non-empty (formatted via `formatToken`).
- No new CSS — reuses existing `assignmentList` + `sectionLabel`.

### Sprint D5: Real-LLM integration smoke
- Smoke against "Airflow → Prefect 3 ETL migration" decision (`MEETKAT_MEMORY_DIR=/tmp/meetkat-d5`). Verified:
  - `runtime_readiness.role_planner_kind == "llm"` and `role_planner_degraded == false` when `.env` provides supplier env.
  - Supervisor outputs `mission: None / deliverable: None` (conductor model intact) and emits real focus_angle hints derived from the requirement (e.g., "Describe current observability: logging depth, alerting coverage, tracing integration, and dashboarding tools").
  - Three specialists produce divergent claims (impl: 87 active DAGs / 63% custom operators; risk: 27% manual recovery / SOC2 CC6.1 audit gap; ops: 42% SLA alerts / no OTel) — clear evidence of LLM-driven authorship, not template output.
  - `mas_room_*.json` contains `latest_supervisor_plan` + `latest_claim.{role}` facts; events include `supervisor.plan` + `specialist.claim` — all journal-anchored.
  - Per-role lessons in `long_term/*.json` are distinct per role.
