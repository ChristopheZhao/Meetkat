## 1. Requirement Analysis
- User intent:
  - 先把 Meetkat 当前的 MAS harness 架构设计清楚，再继续实现。
  - 后续实现必须严格遵守统一架构，不允许 `Runtime/State/Adapter` 再次漂成顶层层名。
  - `memory` 是核心基底，`state` 只是基于 memory 的映射。
  - adapter/helper 不能膨胀成竞争主线。
- Scope in:
  - 仅覆盖 MAS 内核重构。
  - 仅覆盖 `Memory / Orchestration / Control / Governance / Runtime-as-harness / Ports-Adapters` 的落地拆分。
  - 仅覆盖 room/session bounded context 的 SoT、projection、控制与编排边界。
- Scope out:
  - 不覆盖前后端交互层重构。
  - 不覆盖 P0 真实试运行证据补齐。
  - 不覆盖产品功能新增。
- Architecture constraints:
  - 必须遵守 `docs/tech/MAS-harness-architecture-baseline.md`。
  - `RoomEventJournal` 是 room/session bounded context 内唯一 SoT。
  - `state` 只能作为 projection 术语存在。
  - `Runtime/Harness` 只能作为执行线束，不得作为顶层层名。
  - `Guardrail/Gating` 归 `Control` 子域。

## 2. Semantic Closure
- 本计划关闭的问题：
  - 将当前混层的 MAS 内核拆回明确职责边界，使后续实现不再围绕错误层名和错误 SoT 漂移。
- 本计划不要求立即关闭的问题：
  - 浏览器联调与前端交互闭环。
  - 生产级基础设施替换。

## 3. Sprint Contracts

### Sprint A1: Memory Foundation
- What:
  - 固化 `RoomEventJournal` / projection / replay / checkpoint 边界，锁死唯一 SoT。
- Why:
  - 不先固化 SoT，后续 Control 和 Orchestration 拆分会继续基于错误事实边界展开。
- Done signals:
  - room 事实只通过单一 journal append 路径进入系统。
  - projection 可由 journal 重建，不再承担事实源职责。
  - 代码与文档中不再把 `state` 视为独立架构层。
- Evidence checklist:
  - append / replay / projection source 有单一清晰出口。
  - snapshot/transcript 可由 journal 重放重建。
  - 未入 journal 的信息不会成为正式 room 事实。
- Negative cases:
  - `RoomSnapshot` 仍可直接作为主写模型。
  - helper/adapter 通过缓存维护旁路状态。

### Sprint A2: Control Consolidation
- What:
  - 收口 ended/retry/timeout/override/fallback eligibility/gating 的单一控制出口。
- Why:
  - 控制语义继续散落，runtime 壳就不会变薄，且 guardrail/fallback 会继续多出口。
- Done signals:
  - 当前执行是否允许继续，有单一控制出口回答。
  - `Guardrail/Gating` 明确作为 `Control` 子域，不单独漂移。
  - 结束条件、拒写、回退准入不再由多个模块分别裁决。
- Evidence checklist:
  - room-level control policy 有单一出口。
  - fallback 保持 disaster-only，不退化为常态 else。
  - override 后行为一致且可验证。
- Negative cases:
  - provider/router 反向决定业务终止条件。
  - ended 后仍有旁路写入。
  - control policy 与 guardrail logic 出现双出口。

### Sprint A3: Orchestration Extraction
- What:
  - 把角色协作、round synthesis、handoff、consensus content 收口到 Orchestration。
- Why:
  - 不把智能主线抽离 runtime 壳，就无法建立稳定的 harness 边界。
- Done signals:
  - Host/Pro/Con/Recorder 的协作语义由 orchestrator 决定，而不是 runtime 壳硬编码。
  - runtime 不再拥有会议内容生成语义。
  - execution 仍作为 orchestration 的执行臂，而不是被 Control 吞并。
- Evidence checklist:
  - room-level orchestrator 有明确收口点。
  - runtime 壳对角色顺序和内容产物的直接知识显著减少。
  - 共识内容生成与“是否结束”判定边界清晰分离。
- Negative cases:
  - runtime 继续直接知道角色业务语义。
  - Control 开始生成内容或决定推理路径。

### Sprint A4: Harness Thinning And Ports/Adapters Cleanup
- What:
  - 将 `runtime` 收缩为薄 harness/coordinator，并压薄 routing/providers/storage adapters。
- Why:
  - 不清掉实现容器与 adapter 的膨胀风险，前 3 个 sprint 的边界会重新污染。
- Done signals:
  - runtime 只负责循环驱动、层间调度、恢复入口、依赖装配。
  - adapter 只保留边界职责，不再携带业务主逻辑。
  - routing/providers/storage 不再长出第二条控制/编排主线。
- Evidence checklist:
  - `room_runtime.py` 变成薄壳或被更薄的 coordinator 替代。
  - provider/routing 只保留边界职责。
  - 端到端行为仍满足基线文档。
- Negative cases:
  - runtime 继续混有 journal append、control 判定、角色编排。
  - adapter 成为 fallback/control/orchestration 的竞争主线。
